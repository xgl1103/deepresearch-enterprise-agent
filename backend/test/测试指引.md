# DeepResearch 测试套件说明文档

## 1. 概述

本文档面向技术评审人和项目维护者，系统阐述 DeepResearch 项目的测试策略、测试架构设计和覆盖范围。

### 1.1 测试规模

| 指标 | 数值 |
|------|------|
| 测试文件 | **18 个**（不含 conftest） |
| 测试用例 | **245 个** |
| 通过率 | 100%（241 通过，4 跳过） |
| 测试类型 | 单元测试 + 集成测试 |
| Mock 策略 | 纯 `unittest.mock`，零额外依赖 |
| 运行耗时 | ~50 秒 |

### 1.2 测试分层（同心圆模型）

项目采用**六层同心圆**测试策略——越内层越聚焦于业务规则，越外层越依赖 Mock 基础设施。

```
            ┌──────────┐
            │KB生命周期 34│  ← 四档模式 + 过滤/衰减/分类TTL
           ┌┴──────────┴┐
           │  集成 45     │  ← 直接调用子图节点函数 + 路由分支
          ┌┴────────────┴┐
          │ 异常专项 22   │  ← 异常层次/Fail-fast（降级计数器测试已跳过）
         ┌┴──────────────┴┐
         │   异步 11       │  ← astep/aacquire/_aembed（纯异步行为测试）
        ┌┴────────────────┴┐
        │   Mock 50         │  ← Mock OpenAI/dashscope/Milvus/requests
       ┌┴──────────────────┴┐
       │   纯函数 79          │  ← Post/Utils/Config/Schema/Cache/Logger/RateLimiter
       └────────────────────┘
```

**层级归属规则**：当某个测试同时符合多层特征时，归入最内层。例如：一个同时用了 `@pytest.mark.asyncio` 和调用子图节点函数的测试，归入"集成"层而非"异步"层。

#### 纯函数层（79 个测试）— 最外层，确定性 I/O

| 测试文件 | 测试数 | 被测模块 | 验证要点 |
|---------|--------|---------|---------|
| `test_configuration.py` | 15 | `configuration.py` | 模型列表加载（env/默认/非法 JSON）、三级优先级、安全边界 |
| `test_tools_and_schemas.py` | 15 | `tools_and_schemas.py` | 6 个 Pydantic Schema 的正常/边界/必填字段校验 |
| `test_utils.py` | 13 | `utils.py` | URL 短链接映射（7 场景）、研究主题获取、最后用户响应 |
| `test_search_cache.py` | 11 | `search_cache.py` | 命中/过期/清理/统计/线程安全（10 线程并发） |
| `test_logger.py` | 10 | `logger.py` | 正常初始化、OSError 兜底、通用异常兜底、多种日志输入 |
| `test_rate_limiter.py` | 8 | `base_agent.py::RateLimiter` | 令牌桶算法、首次不限、间隔内限流、线程安全（20 线程） |

#### Mock 层（50 个测试）— 模拟外部 SDK 边界

|---------|--------|---------|---------|

| `test_fact_store.py` | 12 | MilvusClient、requests.post | 写入链路（嵌入→插入）、查询（正常/空/置信度过滤）、嵌入 429 重试、集合自动创建 |

| `test_fact_extractor.py` | 8 | Agent | 事实提取正常流程、短文本跳过、校验过滤、10 条上限、非法 JSON 兜底 |

#### 异步层（11 个测试）— 纯异步行为验证

| 测试文件 | 测试数 | 验证要点 |
|---------|--------|---------|
| `test_async_agent.py` | 9 | `RateLimiter.aacquire()`（不限/限流/间隔）、`Agent.astep()`（成功/瞬时重试/PermanentError 快速失败/全部失败返回空）、`WebSearchAgent.astep()`（成功/限流重试） |
| `test_async_fact_store.py` | 2 | `FactStore._aembed()` 同步包装、错误传播 |

**注意**：全项目共 34 个测试标记了 `@pytest.mark.asyncio`，但其中 23 个被上层（集成层/异常专项层）认领。本层仅包含**纯异步行为**测试——不涉及子图节点函数调用。

#### 异常专项层（26 个测试）

| 测试文件 | 测试数 | 验证要点 |
|---------|--------|---------|
| `test_exceptions.py` | 26 | 17 个异常类的继承层级（isinstance 链验证）、`is_transient`/`is_permanent` 分类、8 种 OpenAI SDK 错误→Agent 异常映射、Agent/WebSearchAgent 的 PermanentError 快速失败（重试 0 次）、KB 降级计数器递增/重置/告警阈值、KB Embedding 的 401/403/400 致命错误分类 |

**技术要点**：异常分类是重试策略的基石——`TransientError`（重试）vs `PermanentError`（快速失败）。`_translate_openai_error` 函数将 OpenAI SDK 的原生异常映射为 Agent 异常体系，是整个 pipeline 容错的核心。

#### 集成层（45 个测试）— 子图节点函数 + 路由分支

| 测试文件 | 测试数 | 被测模块 | 验证要点 |
|---------|--------|---------|---------|
| `test_writer_agent.py` | 18 | `sub_agents/writer_agent.py` | 提纲生成、草稿撰写/修订、审稿（有/无问题）、路由决策（5 分支）、终审润色 + URL 替换、**URL 替换边界**（空来源/重复来源/不存在于原文/多次出现） |
| `test_research_agent.py` | 17 | `sub_agents/research_agent.py` | 查询生成（无 KB/有 KB 命中/KB 降级）、扇出、搜索汇总（正常/空结果）、反思评估、路由决策（充分/达上限/继续）、**KB 存储边界**（存储成功/KB 为 None/存储异常静默） |
| `test_graph.py` | 10 | `graph.py` | 图拓扑、计划生成（未确认/已确认跳过）、路由决策（4 分支：等待确认/关键词直通/LLM 隐式确认/LLM 拒绝→重规划）、replan 循环 |

**技术要点**：集成层不启动完整的 LangGraph 图执行引擎，而是直接调用子图节点函数（`_generate_queries`、`_web_search`、`_critique` 等）和路由函数（`_route_after_critique`、`_route_after_critic`、`evaluate_plan`）。这样既验证了完整的逻辑链路，又避免了图执行引擎的异步调度复杂性。

#### KB 生命周期层（34 个测试）— 最内层，业务规则穷举

| 测试文件 | 测试数 | 验证要点 |
|---------|--------|---------|
| `test_kb_lifecycle.py` | 34 | 4 种模式（off/inform/freshness/lifecycle）× 4 个谓词（should_filter/should_decay/should_tag/should_warn）= 16 种组合穷举、FRESHNESS_MAX_AGE 阈值（high=7d/medium=30d/low=180d）、CATEGORY_TTL 按类别 TTL（market_data=7d/product=30d/strategy=90d/technology=180d/historical=永不）、事实存储时 fact_category 字段正确写入 |

**技术要点**：这是项目中**测试密度最高**的模块——34 个测试覆盖 55 行源码。KB 生命周期控制了知识库中事实的新鲜度过滤策略，是"知识库随时间演化"这一核心特性的规则引擎。每个模式组合的穷举测试确保了规则矩阵的完整性。

## 2. 测试基础设施

### 2.1 conftest.py — 共享 Fixtures

所有测试通过 `conftest.py` 获得一致的基础环境，无需每个文件重复配置。

**环境 Fixture（autouse）**

`mock_env` 为每个测试自动注入统一的模拟环境变量，确保测试隔离且可重现：

```python
APP_TOKEN          = "test-api-key-12345"
LLM_BASE_URL       = "https://test-llm.example.com/v1"
MCP_APP_ID         = "test-mcp-app-id"
WEB_SEARCH_MAX_QPS = "100"          # 高 QPS，测试不阻塞
MILVUS_URI         = "http://localhost:19530"
```

设计要点：每个测试都在干净的环境中启动，`autouse=True` 省去重复声明。

**数据 Fixtures**

| Fixture | 用途 | 被使用于 |
|---------|------|---------|
| `sample_messages` | 包含一条 HumanMessage 的消息列表 | Agent 测试、子图测试 |
| `sample_state` | 完整的 OverallState 字典（17 字段） | 所有子图节点测试 |
| `sample_web_results` | 模拟的搜索结果列表 | WebSearchAgent 测试 |

**Mock Fixtures**

| Fixture | Mock 目标 | 设计原因 |
|---------|----------|---------|
| `mock_openai_client` | `agent.llm.llm.OpenAI` | 拦截 LLM API 调用，返回可控响应 |
| `mock_dashscope_app` | `agent.base_agent.Application` | 拦截 MCP 搜索请求 |
| `mock_openai_response` | 工厂函数，生成不同内容的 mock 响应 | 灵活模拟 LLM 输出 |

**Mock 策略原则**：Mock 在"外部边界"——OpenAI SDK、dashscope SDK、requests、MilvusClient。内部逻辑（Agent、JsonAgent、缓存、限流）全部真实执行。

### 3.4 test_configuration.py — 配置系统

**被测模块**: `agent/configuration.py`

**核心测试场景**:

```
test_env_var_takes_precedence  ← env > configurable
    │
    ├── test_configurable_overrides  ← configurable 覆盖默认值
    │
    ├── test_empty_config_returns_defaults  ← 无配置时的默认值
    │
    ├── test_partial_override  ← 部分字段覆盖，其余保持默认
    │
    └── test_available_models_not_overridable_by_configurable  ← 安全边界
```

**技术要点**：`Configuration.from_runnable_config()` 实现了**三级优先级**：环境变量 > configurable 字典 > Pydantic 默认值。这是 LangGraph Agent 的标准配置模式，允许运维通过环境变量统一管理模型选择，同时允许 API 调用方通过 `configurable` 动态调整。

特别重要的测试是 `test_available_models_not_overridable_by_configurable`：防止攻击者通过 configurable 注入恶意模型列表，available_models 只能从服务端环境变量加载。

### 3.5 test_search_cache.py — 搜索缓存

**被测模块**: `agent/search_cache.py`

**测试覆盖**:

| 测试类 | 验证行为 | 关键技术 |
|--------|---------|---------|
| `TestCacheMiss` | MISS/不同 count/不同 prompt | MD5 键生成正确性 |
| `TestCacheHit` | SET→GET 命中 / prompt.strip() 规范化 | 写入后立即可读 |
| `TestCacheExpiry` | TTL 过期后返回 None，entry 被清理 | 时间戳模拟 |
| `TestClearCache` | 清空后所有 entry 不可达 | 批量清理 |
| `TestCacheStats` | 空/非空/过期后统计 | 监控可观测性 |
| `TestCacheThreadSafety` | 10 线程并发写入 | `threading.Lock` 有效性 |

**技术要点**：缓存键使用 `hashlib.md5(prompt.strip():{count})` 生成，带 1 小时 TTL。线程安全通过 `threading.Lock` 保护共享字典。这是项目中唯一有**并发安全验证**的组件——10 线程并发写入后验证每条数据完整可读。

### 3.6 test_rate_limiter.py — 速率限制器

**被测模块**: `agent/base_agent.py::RateLimiter`

**设计验证**:

```
令牌桶算法核心：min_interval = 1.0 / max_qps

test_rapid_acquires_get_throttled:
  1st acquire → wait=0（首次不限）
  2nd acquire → wait>0（距上次 < min_interval）

test_acquire_after_interval_no_wait:
  1st acquire → 模拟 1 秒后
  2nd acquire → wait=0（已过 min_interval）
```

**线程安全验证**: 20 线程同时 acquire，高 QPS（1000）下无死锁，无数据竞争。

**单例验证**: `get_web_search_rate_limiter()` 全局唯一实例，从 `WEB_SEARCH_MAX_QPS` 环境变量读取配置。

### 3.8 test_base_agent.py — Agent 类层次

**被测模块**: `agent/base_agent.py`

这是测试套件中最复杂的文件之一，覆盖了 4 层类继承体系。重试逻辑已统一通过 `_retry_with_classified_errors` 函数实现，测试验证各 Agent 类的正确委托。

#### Agent（基类）— 5 个测试

```
test_step_calls_llm_and_returns_content
  └── Mock OpenAI → 验证 response 透传

test_step_retries_on_failure
  └── side_effect: [Exception, Exception, "success"]
  └── 验证重试后最终成功

test_step_returns_empty_on_all_failures
  └── 3 次全部失败 → 返回 ""

test_prompt_format_replaces_placeholders
  └── "{name}, {age}" → "Alice, 30"

test_prompt_format_missing_key_ignored
  └── "{name}" + other=value → "{name}" 不替换
```

**技术要点**: Agent 的 `step()` 方法是整个 pipeline 的基石。重试逻辑委托给 `_retry_with_classified_errors` 辅助函数——`TransientError` 自动重试，`PermanentError` 快速失败。测试验证了委托正确性和兜底返回空字符串。

#### JsonAgent — 3 个测试

验证了 JSON 提取 → Pydantic 解析的完整流程。`test_post_process_without_keys_returns_dict` 验证了无 keys 参数时返回原生 dict。

#### MCPAgent — 3 个测试

验证了 `dashscope.Application.call` 的调用、post_process 的 JSON 解析、以及解析失败时的重试逻辑。API Key 优先使用 `DASHSCOPE_API_KEY`，fallback 到 `APP_TOKEN`。

#### WebSearchAgent — 5 个测试

| 测试 | 关键验证 |
|------|---------|
| `test_step_returns_processed_pages` | 完整流程：API调用→多路径解析→字段提取 |

### 3.14 test_graph.py — 主编排图集成测试

**被测模块**: `agent/graph.py`

**Plan 阶段测试**（`TestGeneratePlan`）:

- `test_generates_plan_for_unconfirmed_status` — 用户首次提交，生成研究计划
- `test_skips_when_already_confirmed` — 用户已确认计划后不再重复生成

**路由决策测试**（`TestEvaluatePlan`）:

```
plan_status="unconfirmed"
  → "awaiting_plan_confirmation"  (Human-in-the-loop 暂停)

plan_status="confirmed" + 用户输入"需求确认"
  → "confirm_plan"  (关键词直接匹配，跳过 LLM 意图检测)

plan_status="confirmed" + 用户输入"这个计划没问题"
  → JsonAgent 解析 → satisfy=True → "confirm_plan"  (LLM 隐式确认)

plan_status="confirmed" + 用户输入"这个计划不行"
  → JsonAgent 解析 → satisfy=False → "replan"

plan_status="confirmed" + plan=None
  → "replan"  (异常边界：无计划 → 回到等待确认)
```

**技术要点**：`evaluate_plan` 是 Human-in-the-Loop 模式的核心路由函数。它采用**两级判断**：
1. 关键词匹配（快速路径）："开始研究"/"需求确认" → 直接进入 Research
2. LLM 意图检测（慢速路径）：JsonAgent 解析用户意图 → PlanReflection.satisfy

## 4. 测试策略总结

### 4.1 确定性代码（纯函数层）— 完全覆盖

`Post.extract_pattern`、`resolve_urls`、`get_research_topic`、`get_last_user_response`、`setup_logger`、`log_request_details`、所有 Pydantic Schema、`Configuration` 模型、`RateLimiter`、`search_cache`。这些函数无副作用、无外部依赖，测试直接断言输入输出。

### 4.2 非确定性代码（LLM 调用）— Mock 层覆盖

所有 Agent 类、`OpenAICompatibleLLM`、FactExtractor。Mock 策略：在 OpenAI SDK / dashscope SDK 边界注入 mock，验证调用次数、参数、重试逻辑。不测试 LLM 输出质量（那是 eval 框架的职责）。

### 4.3 外部服务（Milvus / Embedding API）— Mock 层覆盖

FactStore 的 `add_facts`、`query`、`_embed` 方法。Mock MilvusClient 和 requests，验证数据流转和错误恢复逻辑。

### 4.4 控制流（路由 / 循环）— 集成层覆盖所有分支

ResearchAgent 的 `_route_after_critique`（3 个分支）、WriterAgent 的 `_route_after_critic`（5 个分支）、主图的 `evaluate_plan`（4 个分支）。

### 4.5 异常处理 — 异常专项层完整覆盖

17 种异常类型的继承层级、8 种 OpenAI SDK 错误映射、PermanentError 快速失败验证、KB 降级计数器。

### 4.6 并发安全 — 压力验证

`test_search_cache.py::TestCacheThreadSafety`（10 线程并发写入）和 `test_rate_limiter.py::TestRateLimiterThreadSafety`（20 线程并发 acquire）。

### 4.7 HTTP 端点 — 新增覆盖

`test_app.py` 验证 `/api/models` 端点（正常/空列表/默认值/非法 JSON）和 HTTP 中间件（GET/POST/异常路径）。`test_logger.py` 验证日志初始化的三条代码路径。
