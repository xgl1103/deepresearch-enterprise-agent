# 企业工程化实施路线

当前阶段继续沿用本地部署形态：前端 Vite、后端 LangGraph/FastAPI、Docker
中的 PostgreSQL、Redis、Milvus。暂不引入 Kubernetes 或云厂商专有服务。

## 推荐实施顺序

1. 任务队列与独立 Worker
2. 鉴权、权限和用户数据隔离
3. 密钥管理与启动配置校验
4. 可观测性（日志、指标、链路追踪、告警）
5. 限流、并发控制和模型成本预算
6. 数据持久化、生命周期和备份恢复
7. 安全审计与自动化评测门禁

## 第一阶段：任务队列与独立 Worker

已实施：

- API 与任务执行可拆分为两个进程；生产模式不在 API lifespan 内执行任务。
- 每个 Worker 使用唯一 Redis consumer name，支持横向增加 Worker。
- 使用 `XAUTOCLAIM` 接管崩溃 Worker 留下的 pending 消息。
- 执行失败最多重试 `TASK_MAX_ATTEMPTS` 次，超过后进入 dead-letter stream。
- Redis 持久化 `queued/running/retrying/paused/completed/cancelled/failed` 状态。
- 状态接口执行与 SSE、取消接口相同的任务所有权校验。
- 本地原有一键启动脚本继续启用内嵌 Worker，避免改变当前开发习惯。

独立模式启动方式：

```powershell
# 终端 1：API（确保 EMBEDDED_TASK_WORKER=false）
run_uvicorn_backend.bat

# 终端 2：Worker
run_worker.bat
```

Linux/macOS Worker：

```bash
./run_worker.sh
```

关键配置见 `backend/.env.example`。生产部署时 API 与 Worker 使用同一套
`REDIS_URL`、模型配置和 MCP 配置，但应作为两个独立的受管进程运行。

## 第二阶段：鉴权与权限隔离

已实施：

- 恢复已有任务前先校验所有权，禁止跨用户复用 `task_id`。
- `thread_id` 全局唯一，并以数据库外键绑定用户；删除用户时级联清理关联。
- LangGraph 原生端点启用自定义认证，并采用 default-deny 授权策略。
- 原生 thread 创建时写入 owner，读、查、改、删、运行均按 owner 过滤。
- Session 同时具有空闲超时和绝对超时，旧格式会话自动失效。
- 密码修改后注销该用户所有会话，防止旧凭证继续访问。
- 不再自动创建 `zhangsan/zhangsan` 等弱口令账号；引导账号必须通过环境变量显式提供，且密码至少 12 位。
- Cookie 的 `Secure` 属性可通过 `SESSION_COOKIE_SECURE=true` 开启，生产 HTTPS 必须开启。

## 第三阶段：密钥管理与配置校验

已实施：

- API 与 Worker 启动时统一校验模型、MCP、Redis、PostgreSQL 配置，缺失立即失败。
- 支持 `APP_TOKEN_FILE`、`MCP_APP_ID_FILE` 等文件型密钥注入，可直接对接 Docker Secrets。
- 同一密钥禁止同时设置明文环境变量和 `*_FILE`，避免来源优先级不清。
- `APP_ENV=production` 时强制安全 Cookie、独立 Worker、非默认数据库密码。
- Docker Compose 不再包含 PostgreSQL 和 MinIO 的硬编码默认口令。
- 根目录 `.env` 已加入 Git 忽略，`.env.docker.example` 仅提供无效占位模板。
- 请求日志对登录密码及新旧密码统一脱敏。

本地启动 Docker 前：

```powershell
Copy-Item .env.docker.example .env
# 编辑 .env，替换全部 replace-with-* 占位值
docker compose up -d
```

企业环境不要把真实密钥写入镜像或 Git；应由进程管理器、CI/CD Secret、
Docker Secret 或后续云密钥服务注入。

## 第四阶段：可观测性、限流与成本控制

已实施：

- 所有 HTTP 响应返回 `X-Request-ID`；合法的上游 ID 会透传，否则服务端生成 UUID。
- Worker 日志自动携带 task ID，API 日志自动携带 request ID。
- 增加 `/health/live` 与 `/health/ready`，ready 会实际检查 Redis 和 PostgreSQL。
- 增加 Prometheus `/metrics`，使用独立 `METRICS_TOKEN` Bearer 认证。
- 指标覆盖 HTTP 数量/耗时/并发、任务状态/耗时/队列、LLM 请求/token/估算成本、配额拒绝。
- 使用 Redis Lua 原子执行每用户分钟、每日及并发任务配额，多 API 实例共享限制。
- 强制限制单任务初始查询数和研究循环数，阻止客户端绕过前端制造无上限成本。
- LLM 层读取供应商 usage；`MODEL_PRICING_JSON` 可按模型配置每百万 token 单价。

关键接口：

```text
GET /health/live
GET /health/ready
GET /metrics     Authorization: Bearer <METRICS_TOKEN>
```

成本估算配置示例（币种由团队统一约定）：

```env
MODEL_PRICING_JSON={"model-id":{"input_per_million":1.0,"output_per_million":4.0}}
```
