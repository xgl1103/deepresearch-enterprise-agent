# DeepResearch 测试套件说明文档

## 1. 概述

本文档面向面试官和技术评审人，系统阐述 DeepResearch 项目的测试策略、测试架构设计和覆盖范围。

### 1.1 测试规模

| 指标 | 数值 |
|------|------|
| 测试文件 | **22 个**（含 conftest） |
| 测试用例 | **297 个** |
| 通过率 | 100%（293 通过，4 跳过） |
| 测试类型 | 单元测试 + 集成测试 |
| Mock 策略 | 纯 `unittest.mock`，零额外依赖 |
| 运行耗时 | ~60 秒 |

### 1.2 测试分层（同心圆模型）

项目采用**六层同心圆**测试策略——越内层越聚焦于业务规则，越外层越依赖 Mock 基础设施。

```
            ┌──────────┐
            │KB生命周期 34│  ← 四档模式 + 过滤/衰减/分类TTL
           ┌┴──────────┴┐
           │  集成 47     │  ← 直接调用子图节点函数 + 路由分支
          ┌┴────────────┴┐
          │ 异常专项 30   │  ← 异常层次/Fail-fast/重试耗尽/配置错误/连接错误
         ┌┴──────────────┴┐
         │   异步 11       │  ← astep/aacquire/_aembed（纯异步行为测试）
        ┌┴────────────────┴┐
        │   Mock 83         │  ← Mock OpenAI/dashscope/Milvus/requests/TextReRank
       ┌┴──────────────────┴┐
       │   纯函数 88          │  ← Post/Utils/Config/Schema/Cache/Logger/RateLimiter/Reranker
       └────────────────────┘
```

**层级归属规则**：当某个测试同时符合多层特征时，归入最内层。例如：一个同时用了 `@pytest.mark.asyncio` 和调用子图节点函数的测试，归入"集成"层而非"异步"层。

#### 异常专项层（30 个测试）

| 测试文件 | 测试数 | 验证要点 |
|---------|--------|---------|
| `test_exceptions.py` | 30（4 跳过） | 17 个异常类的继承层级（isinstance 链验证）、`is_transient`/`is_permanent` 分类、8 种 OpenAI SDK 错误→Agent 异常映射、Agent/WebSearchAgent 的 PermanentError 快速失败（重试 0 次）、KB Embedding 的 401/403/400 致命错误分类、KB Embedding 瞬时错误（429/5xx/网络/未知）重试耗尽→`KBEmbeddingError`、KB 配置错误（缺少 URL/API Key）→`KBConfigError`、Milvus 连接失败→`KBConnectionError`、KB 集合维度参数错误→`KBConfigError` |
| `test_reranker.py` | 7 | 重排器错误处理：HTTP 500/401/429退避降级/网络错误/意外异常/None 输出 |

**技术要点**：异常分类是重试策略的基石——`TransientError`（重试）vs `PermanentError`（快速失败）。`_translate_openai_error` 函数将 OpenAI SDK 的原生异常映射为 Agent 异常体系，是整个 pipeline 容错的核心。KB 异常体系现已完整落地：`KBConnectionError`（Milvus 连接失败）、`KBEmbeddingError`（Embedding 瞬时错误重试耗尽）、`KBEmbeddingFatalError`（Embedding 认证/参数/解析失败）、`KBConfigError`（配置缺失或错误）。

#### 集成层（47 个测试）— 子图节点函数 + 路由分支

| 测试文件 | 测试数 | 被测模块 | 验证要点 |
|---------|--------|---------|---------|
| `test_research_agent.py` | 21 | `sub_agents/research_agent.py` | 图拓扑、查询生成（无 KB/有 KB 命中/KB 降级）、扇出、搜索汇总（正常/空结果/重排器启用/降级）、反思评估、路由决策（充分/达上限/继续）、KB 存储边界（存储成功/KB 为 None/存储异常静默） |
| `test_writer_agent.py` | 13 | `sub_agents/writer_agent.py` | 图拓扑、提纲生成、草稿撰写/修订、审稿（有/无问题）、路由决策（4 分支）、终审润色 + URL 替换 + 来源去重 |
| `test_graph.py` | 13 | `graph.py` | 图拓扑、计划生成（未确认/已确认跳过）、路由决策（4 分支：等待确认/关键词直通/LLM 隐式确认/LLM 拒绝→重规划）、replan 循环 |

**技术要点**：集成层不启动完整的 LangGraph 图执行引擎，而是直接调用子图节点函数（`_generate_queries`、`_web_search`、`_critique` 等）和路由函数（`_route_after_critique`、`_route_after_critic`、`evaluate_plan`）。这样既验证了完整的逻辑链路，又避免了图执行引擎的异步调度复杂性。

#### KB 生命周期层（34 个测试）— 最内层，业务规则穷举

| 测试文件 | 测试数 | 验证要点 |
|---------|--------|---------|
| `test_kb_lifecycle.py` | 34 | 4 种模式（off/inform/freshness/lifecycle）× 4 个谓词（should_filter/should_decay/should_tag/should_warn）= 16 种组合穷举、FRESHNESS_MAX_AGE 阈值（high=7d/medium=30d/low=180d）、CATEGORY_TTL 按类别 TTL（market_data=7d/product=30d/strategy=90d/technology=180d/historical=永不）、Milvus 层面新鲜度过滤集成验证、事实存储时 fact_category 字段正确写入 |

**技术要点**：这是项目中**测试密度最高**的模块——34 个测试覆盖 55 行源码。KB 生命周期控制了知识库中事实的新鲜度过滤策略，是"知识库随时间演化"这一核心特性的规则引擎。每个模式组合的穷举测试确保了规则矩阵的完整性。

## 2. 测试基础设施

### 2.1 conftest.py — 共享 Fixtures

所有测试通过 `conftest.py` 获得一致的基础环境，无需每个文件重复配置。

**环境 Fixture（autouse）**

`mock_env` 为每个测试自动注入统一的模拟环境变量，确保测试隔离且可重现：

```python
APP_TOKEN          = "test-api-key-12345"
LLM_BASE_URL       = "https://test-llm.example.com/v1"
MCP_APP_ID         = "test-mcp-app-id"
WEB_SEARCH_MAX_QPS = "100"          # 高 QPS，测试不阻塞
MILVUS_URI         = "http://localhost:19530"
EMBEDDING_MODEL    = "text-embedding-v3"
# RERANKER_KB_ENABLED / RERANKER_WEB_ENABLED 默认设为 "false"
```

设计要点：每个测试都在干净的环境中启动，`autouse=True` 省去重复声明。会主动删除 `NUMBER_OF_INITIAL_QUERIES`、`MAX_RESEARCH_LOOPS` 等可选 key 以防止残留干扰。

**数据 Fixtures**

| Fixture | 用途 | 被使用于 |
|---------|------|---------|
| `sample_messages` | 包含一条 HumanMessage 的消息列表 | Agent 测试、子图测试 |
| `sample_state` | 完整的 OverallState 字典（17 字段） | 所有子图节点测试 |
| `sample_web_results` | 模拟的搜索结果列表 | WebSearchAgent 测试 |

**Mock Fixtures**

| Fixture | Mock 目标 | 设计原因 |
|---------|----------|---------|
| `mock_openai_client` | `agent.llm.llm.OpenAI` | 拦截 LLM API 调用，返回可控响应 |
| `mock_openai_response` | 工厂函数，生成不同内容的 mock 响应 | 灵活模拟 LLM 输出 |
| `mock_dashscope_app` | `agent.base_agent.Application` | 拦截 MCP 搜索请求 |

**Mock 策略原则**：Mock 在"外部边界"——OpenAI SDK、dashscope SDK、requests、MilvusClient、redis.asyncio。内部逻辑（Agent、JsonAgent、缓存、限流）全部真实执行。

### 2.2 运行方式

```bash
# 安装依赖
cd backend && pip install -e ".[dev]"

# 运行全部测试
python -m pytest test/ -v

# 按模块运行
python -m pytest test/test_base_agent.py -v
python -m pytest test/test_reranker.py -v
python -m pytest test/test_task_queue.py -v

# 按标记运行
python -m pytest test/ -m unit -v
python -m pytest test/ -m integration -v

# 带覆盖率报告
python -m pytest test/ --cov=src/agent --cov-report=html

# 排除异步测试（调试时快速反馈）
python -m pytest test/ -v -m "not asyncio"
```

## 5. 测试策略总结

### 5.1 确定性代码（纯函数层）— 完全覆盖

`Post.extract_pattern`、`resolve_urls`、`get_research_topic`、`get_last_user_response`、`setup_logger`、`log_request_details`、所有 Pydantic Schema、`Configuration` 模型、`RateLimiter`、`search_cache`、`RerankerService`（初始化/配置部分）。这些函数无副作用、无外部依赖，测试直接断言输入输出。

### 5.2 非确定性代码（LLM 调用）— Mock 层覆盖

所有 Agent 类、`OpenAICompatibleLLM`、`FactExtractor`。Mock 策略：在 OpenAI SDK / dashscope SDK / DashScope TextReRank 边界注入 mock，验证调用次数、参数、重试逻辑。不测试 LLM 输出质量（那是 eval 框架的职责）。

### 5.3 外部服务（Milvus / Embedding API / Redis）— Mock 层覆盖

`FactStore` 的 `add_facts`、`query`、`_embed` 方法。Mock MilvusClient 和 requests，验证数据流转和错误恢复逻辑。`task_queue` 的 Redis Streams 操作使用 `AsyncMock` 验证读写和终止事件。

### 5.4 控制流（路由 / 循环）— 集成层覆盖所有分支

ResearchAgent 的 `_route_after_critique`（3 个分支）、WriterAgent 的 `_route_after_critic`（4 个分支）、主图的 `evaluate_plan`（4 个分支）。

### 5.5 异常处理 — 异常专项层完整覆盖

17 种异常类型的继承层级、8 种 OpenAI SDK 错误映射、PermanentError 快速失败验证、重排器 5 种 HTTP 错误处理（含 429 退避和降级）。

### 5.6 并发安全 — 压力验证

`test_search_cache.py`（10 线程并发写入）和 `test_rate_limiter.py`（20 线程并发 acquire）。

### 5.7 HTTP 端点 — 覆盖

`test_app.py` 验证 `/api/models` 端点（正常/空列表/默认值/非法 JSON）和 HTTP 中间件（GET/POST/异常路径）。
