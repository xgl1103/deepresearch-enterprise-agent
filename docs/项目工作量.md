## 七、关键假设与说明

1. **估算基于 1 名全栈工程师**独立开发，部分阶段可能有 code review 协助
2. **学习成本已包含**：LangGraph、Milvus、Redis Streams、SSE 等技术的调研和学习时间计入架构设计阶段
3. **不包含以下工作**：
   - CI/CD 流水线搭建（无基础设施）
   - 性能测试与压测（无环境）
   - 安全审计（内部使用）
   - 多语言/国际化（仅中文）
   - 生产环境部署与运维（未上线）
4. **开发周期约 11 个月**但并非全职投入——估算约 10.25 人月的净工作量，考虑到企业内部项目可能有其他事务穿插，实际日历时间与净工作量的比例约 1.1:1

## 八、如果要复制此项目

对于有类似技术栈经验（Python + LangGraph + React）的团队：

| 团队规模 | 预估周期 | 说明 |
|----------|----------|------|
| 1 人全栈 | 10-12 个月 | 需同时覆盖前后端，测试和文档可能被压缩 |
| 2 人（1 后端 + 1 前端） | 5-6 个月 | 后端约 9 PM + 前端约 1.1 PM，可并行 |
| 3 人（2 后端 + 1 前端） | 4-5 个月 | 后端 Agent 和 KB 可并行开发，一个负责流水线一个负责基础设施 |

**加速前提**：团队已有 LangGraph 经验 + Milvus/Redis 运维经验 + 企业内部 LLM API 已就绪。

**最大风险点**（按延期概率排序）：
1. LLM 输出不稳定 → JSON 解析和 Prompt 调优反复迭代（`post.py`、`_safe_format` 等全是这类问题的累积）
2. LangGraph 版本 Breaking Changes → `subgraphs=True` 事件格式、checkpointer API 变更需预留适配时间
3. Redis Streams 任务队列 → Consumer Group 创建冲突、SSE 断线重连、`XACK` 时机调试较耗时
4. KB 生命周期模式 → 4 种模式 × 多个维度（3 fresh_level × 5 category TTL）的测试组合
5. 交叉编码器精排 → fakestore/milvus 中的 reranker 候选补充 + 三重过滤（相关性/置信度/时效）逻辑复杂
